3.制作镜像

  • 不同的需求，例如在部署静态HTML文件之前，需要在系统上配置并运行NGINX，这时，就需要重新制作镜像
  • 建议使用特定的标记而不是latest
  • 创建镜像后需要配置镜像，使用COPY(将本地文件放到镜像中)或者RUN这样的命令
  • 将容器向外暴露端口 使用EXPOSE : EXPOSE 80 433 or EXPOSE 7000-8000
  • 定义启动应用命令，Dockerfile中的CMD行定义了在启动容器时要运行的默认命令 for example: [“cmd”, “-a”, “arga value”, “-b”, “argb-value”] 则启动命令为cmd -a arga value -b argb-value；CMD的另一种方法是ENTRYPOINT，容器启动时，CMD将会被覆盖，ENTRYPOINT定义命令，可以将参数传递给它
  • docker build -t name .
#在Dockerfile中
FROM nginx:1.11-alpine
COPY index.html /usr/share/nginx/html
EXPOSE 80
CMD nginx -g daemon off;

4.部署一个Node.js应用

  • Node.js为每一个版本提供预先构建的有标签的镜像 Node 7.0就是node:7-alpine(这是一种基于阿尔基的构建，它比官方图像更小、更流畅)

  • RUN 执行例如mkdir /src/app/可以创建一个环境作为用户可以读写

  • WORKDIR /src/app/ 以确保将来的所有命令都从相对于我们的应用程序的目录执行

  • 如何安装运行程序的依赖，对于Node.js来说需要NPM

  • 为了将构建时间保持在最低限度，Docker缓存在Dockerfile中执行行的结果，以便在将来的构建中使用，会实时更新

  • 使用NPM我们只想重新运行npm install，如果我们的package.json文件中的内容已经改变了。如果没有任何改变，我们可以使用缓存版本来加快部署

  • 用COPY package.json 如果package.json文件已经改变，我们可以使RUN npm install命令失效。如果文件没有改变，那么缓存将不会被取消，并且会使用npm install命令的缓存结果

  • 如果不想让缓存称为镜像的一部分，可以在docker build 中添加-no-cache=true选项

  • 在我们安装了我们的依赖之后，我们想要复制应用程序的其他源代码。拆分依赖项的安装并复制源代码使我们能够在需要时使用缓存；如果我们在运行npm install之前复制我们的代码，那么它会每次运行，因为我们的代码已经改变了。通过复制package.json，我们可以确定，只有当我们的包内容已经改变时，缓存才会失效

  • 将所有文件放到/src/app/ 设置端口 EXPOSE 3030

  • 最后，应用程序需要启动。使用Node.js的一个巧妙的技巧是使用npm start命令。这看起来在package.json文件知道如何启动应用程序保存命令的重复

#Dockerfile
# This is your Editor pane. Write the Dockerfile here and 
# use the command line to execute commands
FROM node:7-alpine
RUN mkdir -p /src/app
WORKDIR /src/app
COPY package.json /src/app/package.json
RUN npm install

COPY . /src/app
EXPOSE 3000
CMD [ "npm", "start" ]

5.优化Dockerfile用OnBuild

  • 虽然Dockerfile的执行顺序是自上而下的，但你可以在稍后的时候触发一个指令，当镜像被用作另一个镜像的基础时，就会被执行。这样就可以延迟运行以依赖你正在创建的镜像
  • 与前面的场景不同，应用程序指定的命令都是在ONBUILD中预先设置的
  • 结果是我们可以构建这个映像，但是在构建映像用作基本映像之前，应用程序特定的命令不会被执行。然后，它们将作为基础映像构建的一部分执行.

FROM node:7
RUN mkdir -p /usr/src/app
WORKDIR /usr/src/app
ONBUILD COPY package.json /usr/src/app/
ONBUILD RUN npm install
ONBUILD COPY . /usr/src/app
CMD [ "npm", "start" ]

FROM node:7-onbuild
EXPOSE 3000

  • 运行一个负载均衡服务器，当一个容器加载的时候可以动态的发现和更新其负载均衡设置叫做nginx-proxy
  • Nginx-proxy接受HTTP请求，并根据请求主机名将请求代理到适当的容器。这对用户来说是透明的，不会有额外的性能开销

19.Docker Swarm

  • 将单个主机Docker主机转换为多主机Docker群组模式。成为Manager默认情况下，Docker作为一个孤立的单一节点。所有容器只能部署到引擎上。 Swarm模式将其变成一个支持多主机群集的引擎
  • 初始化Swarm模式的第一个节点成为管理者。随着新节点加入集群，他们可以调整管理者或工作者之间的角色。您应该在生产环境中运行3-5名管理，以确保高可用性

  • 第一件事是启用新的实验性标志，以启用指标功能并定义指标地址以在localhost：4444上进行监听

[root@host01 ~]# sed -i 's/dockerd/dockerd --experimental --metrics-addr localhost:9323/g' /etc/systemd/tem/docker.service
[root@host01 ~]# systemctl daemon-reload
[root@host01 ~]# systemctl restart docker

  • 1.定义间隔时间 2.定义了Prometheus应该从中获取数据的服务器和端口。在这个例子中，我们定义了三个不同的目标
  • 9090是普罗米修斯本身。普罗米修斯提现与其内部指标和表现有关的信息；9100是节点到普罗米修斯过程。这暴露了有关节点的信息，例如磁盘空间，内存和CPU使用情况；9323是刚刚暴露的Docker度量标准端口。这从Docker引擎报告信息，更配置信息参考

          group: 'prometheus'
