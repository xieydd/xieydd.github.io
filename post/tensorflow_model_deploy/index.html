[部署TensorFlow模型]

来自于Vitaly Bezgachev

1.第一部分:准备模型

Tensorflow是为生产而生，所以Tensorflow提供解决方案Tensorflow Serving,基本流程，创建模型 -> 本地测试 -> 创建web服务 -> 使用web服务创建部署容器 -> 测试容器 ->放入生产，具体步骤如下: - 使用TensorFlow创建模型 - 使用Docker来镜像化 - 使用TensorFlow Serving托管模型 - 部署到生产环境的Kubernetes

这里使用Udacity深度学习基础课程中的GAN Model for Semi-Supervised learning,这里我们将做以下几个方面: - 在The Street View House Numbers (SVHN) Dataset数据集上训练GAN Model for Semi-Supervised - 使用GAN discriminator预测门牌号，即0-9的数对应的分数 - 将Tensorflow构建的模型集成到Docker容器中 - 创建客户端，提交数字照片 - 部署到云上

基于Jupyter notebook，将功能集成到python文件里:测试保存的模型、模型的输出(使用Protobuf格式)、客户端的服务请求，具体细节见

Tensorflow Serving
  • 实现GRPC接口，所以不能直接发送请求，而是创建一个通过GRPC通信的客户端
  • TensorFlow Serving 已经提供将模型转换成Protobuf的操作
  • 也可以创建自己的模型转换格式实现将其转换成其他格式，但是grpc很快

Tensorflow Serving提供类为SavedModelBuilder

这个模型的输入为(batch_num,width,height,channels)的张量图片为32x32x3且像素点缩放到(-1,1);服务必须接受JPEG图像

#图像转换
#序列化传入图像占位符
serialized_tf_example = tf.placeholder(tf.string,name ='input_image')

feature_configs = {'image/encoded'：tf.FixedLenFeature(shape = [],dtype = tf.string),}

#得到图像张量
jpegs = tf_example ['image/encoded'] 
images = tf.map_fn(preprocess_image,jpegs,dtype = tf.float32) 
images = tf.squeeze(images,[0])
#现在形状是[1,x,x,3]

使用SavedModelBuilder方法将模型转换成Protobuf格式

builder = tf.saved_model.SavedModelBuilder(export_path)

你必须用输入、输出、和(预测还是分类)创建签名，TensorFlow提供了一个方法tf.saved_model.utils.build_tensor_info来创建张量信息

predict_tensor_inputs_info = tf.saved_model.utils.build_tensor_info(jpegs)
predict_tensor_scores_info = tf.saved_model.utils.build_tensor_info(net.discriminator_out)

prediction_signature = (
    tf.saved_model.signature_def_utils.build_signature_def(
    inputs = {"images":predict_tensor_inputs_info},
    outputs = {"scores":predict_tensor_scores_info},
    mether_name = tf.saved_model.signature_constants.PREDICT_METHOD_NAME
    )
)

在教程中创建两个签名：分类和预测，最后保存模型

legacy_init_op = tf.group(tf.tables_initializer(),name = 'legacy_init_op')
builder.add_meta_graph_and_variables(
    sess,
    [tf.saved_model.tag_constants.SERVING],
    signature_def_map = {'predict_images'：prediction_signature},
    legacy_init_op = legacy_init_op
)
builder.save()

这样模型被存储为Protobuf格式，导出结构如下: - 变量文件夹variables.data-xxx-of-yyy and variables.index - saved_model.pb 文件

2.第二部分:容器化

Tensorflow Serving将处理传入的请求转发给模型的服务器，该服务器可能在公有云（例如Amazon AWS，Google云平台，Microsoft Azure）上运行

通常的做法是将这样的服务器和相关的依赖全部打包部署。但是这样的一个虚拟机镜像是很重的。这里我们使用容器创建可部署的构件，这样我们可以部署到本地，内网，以及云上。这样的容器将包括你的服务和所有的依赖，您只需要安装在操作系统上并运行容器，我们会现在本地进行测试。

注意 - Docker命令使用sudo(也可以直接使用docker)，添加docker组，将$User加入到docker组中,然后从其电脑

$ sudo group add docker
$ sudo usermod -aG docker $USER
创建Tensorflow Serving容器
  • 获取Tensorflow Serving
#不但可以获得TensorFlow服务来源，还可以获得TensorFlow本身和TensorFlow模型作为子模块
$ cd〜
$ git clone --recurse-submodules https://github.com/tensorflow/serving.git
  • Tensorflow Serving提供两个Dockerfile用于创建镜像，一个用于CPU构建Dockerfile.devel，一个用于GPU构建Dockerfile.devel-gpu，在/ tensorflow_serving/tools/docker中，这里我们注释掉了Bazel构建
# Build TensorFlow Serving and Install it in /usr/local/bin
WORKDIR /serving
#RUN bazel build -c opt — config=cuda \
# --crosstool_top=@local_config_cuda//crosstool:toolchain \
# tensorflow_serving/model_servers:tensorflow_model_server && \
# cp bazel-bin/tensorflow_serving/model_servers/tensorflow_model_server /usr/local/bin/ && \
# bazel clean --expunge
CMD [“/bin/bash”]

注意:接下来使用的都是GPU，也可以修改并创建CPU，方式相同

$ cd serving
$ docker build --pull -t $USER/tensorflow-serving-devel-gpu -f tensorflow_serving/tools/docker/Dockerfile.devel-gpu .

将花费10-20分钟下载构建镜像的依赖

$ docker run --name=tf_container_gpu -it $USER/tensorflow-serving-devel-gpu

这样就可以进入TensorFlow Serving Docker container的命令行中；注意:当使用exit退出后可以docker start -i tf_container_gpu来重新启动，但是不要移除镜像，这样会导致你做的改变都失效

  • 在镜像中，所需要的组件例如python和Bazel等都已经安装好
在构建好的容器中，构建并启动Tensorflow Serving

    创建客户端

  • 这里在容器中创建客户端，这里我们在本机上创建客户端，向Tensorflow Serving托管的模型发送请求，注意这里我们已经下TensorFlow Serving repository

$ pip install grpcio grpcio-tools
  • 对于我们的客户端需要TensorFlow服务API，但是API是作为Protobuf提供的(它可以在服务/tensorflow_serving/apis下找到)如果我们想在客户端使用它们，我们需要从这些Protobuf生成Python文件。

如果自己生成，如下:

# 1
$ cd <tensorflow serving source folder>
# 2 TensorFlow Serving使用核心框架Protobuf（服务/张量流/张量流/核心/框架），但是服务中定义的导入路径不正确。所以我不得不将tensorflow/tensorflow文件夹移动一层
$ mv ./tensorflow ./tensorflow_
$ mv ./tensorflow_/tensorflow .
# 3 我们从Protobuf为我们的客户端生成Python文件
$ python -m grpc.tools.protoc ./tensorflow_serving/apis/*.proto --python_out=<path to GAN project> --grpc_python_out=<path to GAN project> --proto_path=.
# 4 从2恢复更改
$ mv ./tensorflow ./tensorflow_
$ mv ./tensorflow_ ./tensorflow
  • 客户端已实现，在这里，您只需使用GRPC和TensorFlow服务API来发出请求。最有趣的部分是:
 # Send request
 with open(FLAGS.image,'rb') as f:
    #1 打开JPEG文件
    data = f.read()
    #2 创建预测请求对象
    request = predict_pb2.PredictRequest()
    
    #3 对我们的GAN模型初始化预测请求，这里需要制定模型的名字(必须和开启tensorflow_model_server时设定的model_name参数相同)
    request.model_spec.name = 'gan' 
    request.model_spec.signature_name = 'predict_images'
    request.inputs['images'].CopyFrom(tf.contrib.util.make_tensor_proto(data, shape=[1]))
    
    #4 调用得到结果
    result = stub.Predict(request,60.0)

3.上云

  • 实现Web服务(这里是预测服务)，第一是将其部署到公有云平台保证可用性；第二自动化，即自动部署、拓展、管理服务，即Kubernetes发挥作用

  • 这里使用的是Microsoft Azure(提供30天的200美元的试用)

    准备Docker镜像

  • 进入kubernetes之前，我们需要将所做修改的容器保存下来，得获取容器号

$ docker ps --all
  • 我不想在云中安装昂贵的GPU驱动的虚拟机，我使用了TensorFlow Serving CPU Docker容器
$ docker commit 35027f92f2f8 $USER/tensorflow-serving-gan:v1.0

使用docker images命令得到新创建的容器$User /ensorflow-serving-gan:v1.0 大概是原来镜像的5倍

Kubernetes部分

  • 这里我们创建一个主节点一个工作节点，POD只运行在工作节点，主节点负责调度
  • Pod内可以有多个容器，这里我们只放TensorFlow GAN Serving容器，创建两个Pod
  • Replication controllers动态控制Pod的创建和销毁，虽然每个Pod都有自己的IP地址，但是是没法进行通信的，这里使用Service接入相同的网络
